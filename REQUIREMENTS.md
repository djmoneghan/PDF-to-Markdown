## Feature 1 — PDF Extraction & Structural Integrity

**Module:** `ingestor/extractor.py`

### What it does
Accepts a path to a PDF file and returns a structured document object containing: text content, identified headers (with level), tables (converted to GFM), detected images (with page reference), and detected formula blocks (preserved as raw LaTeX).

### Acceptance Criteria

**AC-1.1 — Docling primary path**
Given a valid PDF path, `extract(pdf_path)` calls Docling's document conversion pipeline and returns a `DocumentContent` dataclass. The function does not call PyMuPDF unless `config.yaml` specifies `engine: pymupdf` or Docling raises an import or runtime error.

**AC-1.2 — Table → GFM conversion**
Every table detected by Docling must be rendered as GitHub Flavored Markdown (GFM) table syntax in the output. Example:
```
| Column A | Column B |
|---|---|
| Value 1  | Value 2  |
```
If table parsing fails for a specific table, raise `ExtractionWarning` with page number and table index. Do not emit raw text or an empty block in place of a failed table.

**AC-1.3 — Formula preservation**
Any content identified as LaTeX or mathematical notation must be wrapped in `$$...$$` blocks, verbatim, with no reformatting. Formula content must never be modified, cleaned, or summarized by the extractor.

**AC-1.4 — Image detection (non-extractive)**
Images are detected and their page numbers and bounding box coordinates are recorded in the `DocumentContent` object. Images are not extracted or embedded in the output Markdown. The chunk containing an image location must include a placeholder: `<!-- IMAGE: page {N}, position {bbox} -->`.

**AC-1.5 — PyMuPDF fallback**
If `engine: pymupdf` is configured, use `fitz` for text extraction. Tables and formulas will not be structurally detected in this mode. The extractor must log a `WARNING` that structural features are unavailable and set `extraction_engine: "pymupdf"` in all chunk metadata for this run.

**AC-1.6 — Unsupported file types**
If the input path does not resolve to a readable PDF, raise `ValueError` immediately with the path and reason. Do not attempt extraction.

## Feature 2 — Semantic Chunking

**Module:** `ingestor/chunker.py`

### What it does
Accepts a `DocumentContent` object and splits it into a list of `Chunk` dataclasses, each representing a coherent topic bounded by header levels. Each chunk carries its breadcrumb trail.

### Acceptance Criteria

**AC-2.1 — Header-based splitting**
Chunks are created at `#` (H1) and `##` (H2) boundaries as configured in `config.yaml` under `chunking.split_levels`. Splitting at `###` or deeper is not performed unless explicitly added to config.

**AC-2.2 — Breadcrumb injection**
Each chunk's `breadcrumb` field is constructed from the header hierarchy at the point of the split. Example: if the chunk lives under `# Section 2` → `## Subsection 2.1`, the breadcrumb is `"Section 2 > Subsection 2.1"`. The `parent_header` field contains only the immediate parent header text.

**AC-2.3 — Token bounds respected**
No chunk is emitted with fewer tokens than `min_chunk_tokens` or more than `max_chunk_tokens` (from config). 
- If a header section is too short: merge it with the next sibling section and log a `DEBUG` message noting the merge.
- If a header section is too long: split at paragraph boundaries (blank lines) until within bounds. If a single paragraph exceeds `max_chunk_tokens`, split mid-paragraph and emit a `WARNING`.

**AC-2.4 — Formula blocks never split**
A `$$...$$` block must never be split across two chunks. If a formula block would cause a chunk to exceed `max_chunk_tokens`, it is kept whole in the chunk it belongs to, and the overflow is accepted with a logged `WARNING`.

**AC-2.5 — Sequential chunk IDs**
Chunks are assigned sequential IDs: `chunk_001`, `chunk_002`, etc., zero-padded to three digits. IDs restart at `001` per document run.

**AC-2.6 — Page range tracking**
Each chunk records the first and last page number of the source PDF it spans, as `page_range: [start, end]`.

## Feature 3 — Metadata Enrichment

**Module:** `ingestor/metadata.py`

### What it does
Accepts a `Chunk` object and calls the local Ollama endpoint to generate the YAML metadata fields that require LLM inference: `summary`, `confidence_score`, and `topic_category`. Assembles the full YAML frontmatter block.

### Acceptance Criteria

**AC-3.1 — Ollama connectivity check**
On first call in a session, perform a health check against `{endpoint}/api/tags`. If unreachable, raise `ConnectionError` with the configured endpoint URL. Do not proceed to metadata generation.

**AC-3.2 — Summary generation**
The `summary` field is generated by prompting the configured model with the chunk's Markdown content. The prompt must instruct the model to produce exactly 1–2 sentences, no bullet points, domain-appropriate language. The raw LLM response is trimmed of whitespace and stored. If the response exceeds 300 characters, log a `WARNING` and truncate at the last sentence boundary within 300 characters.

**AC-3.3 — Confidence score**
The `confidence_score` is generated by a separate prompt asking the LLM to self-assess extraction quality on a 0.0–1.0 scale based on the chunk content (e.g., presence of garbled text, incomplete tables, suspicious formula rendering). The score must be parsed as a float. If parsing fails, default to `0.5` and log a `WARNING`.

**AC-3.4 — Topic category (controlled vocabulary)**
The `topic_category` field must be drawn from this controlled list. The LLM is prompted to select the single best match:

```
- Fuel Cycle
- Reactor Design
- Construction
- Advanced Manufacturing
- Storage, Transportation, and Disposal
- Radioisotopes
- Materials Science
- Thermal Hydraulics
- Instrumentation & Control
- General Reference
```

If the LLM returns a value not on this list, log a `WARNING` and default to `"General Reference"`. Do not use the non-conforming value.

**AC-3.5 — Technical level assignment**
`technical_level` is assigned by the LLM selecting from: `Executive`, `Specialist`, `PhD`. Same controlled vocabulary enforcement as AC-3.4.

**AC-3.6 — Prompt isolation**
Summary, confidence, topic category, and technical level are generated in **separate Ollama calls**, not a single combined prompt. This prevents one field's failure from corrupting others and enables targeted retry.

**AC-3.7 — Retry on timeout**
Each Ollama call retries once on timeout (configurable `timeout_seconds`). If the second attempt also times out, raise `MetadataGenerationError` with the chunk ID and field name.

**AC-3.8 — Full YAML schema compliance**
The assembled YAML block must conform exactly to the schema defined in `CLAUDE.md`. Missing fields are a hard failure. The YAML is validated against the schema before the chunk is passed to the exporter.

## Feature 4 — HITL Review Interface

**Module:** `ingestor/hitl.py`

### What it does
Presents each chunk to the user in a terminal UI showing the Markdown content and proposed YAML side-by-side. The user can Accept, Edit, or Flag each chunk. Edits and flags are recorded to `corrections.json`.

### Acceptance Criteria

**AC-4.1 — Side-by-side layout**
Using Rich `Columns` or `Layout`, display:
- Left panel: rendered Markdown content of the chunk (Rich `Markdown` renderer)
- Right panel: the proposed YAML frontmatter as a syntax-highlighted code block

Both panels must be visible simultaneously without scrolling for chunks under ~40 lines. Longer chunks scroll within their panel.

**AC-4.2 — Three-action interface**
After displaying a chunk, the user is prompted:
```
[A] Accept   [E] Edit   [F] Flag for re-extraction
```
Implemented via `prompt_toolkit` `prompt()`. Accepts single-keystroke input (no Enter required). Invalid keystrokes are ignored and the prompt re-displays.

**AC-4.3 — Auto-accept above threshold**
If `confidence_score` ≥ `hitl.auto_accept_above` (from config), the chunk is automatically accepted without displaying the HITL interface. A single-line Rich status message is printed: `✓ Auto-accepted chunk_NNN (confidence: 0.94)`.

**AC-4.4 — Edit mode**
If the user selects Edit, open the YAML block in an inline `prompt_toolkit` buffer pre-populated with the current YAML. The user edits and confirms with `Ctrl+D`. The edited YAML is re-validated against the schema (AC-3.8 logic). If invalid, display the validation error and re-open the editor. On successful edit, set `hitl_status: "edited"`.

**AC-4.5 — Flag mode**
If the user selects Flag, the chunk is saved with `hitl_status: "flagged"` and is written to the output directory. A record is added to `corrections.json` with `action: "flagged"`, `chunk_id`, `timestamp`, and `reason: null` (user may optionally type a reason before confirming the flag).

**AC-4.6 — Progress display**
A Rich `Progress` bar is displayed at the bottom of the terminal showing `Chunk N of M` throughout the review session.

**AC-4.7 — Session interruption**
If the user sends `Ctrl+C`, the session halts gracefully. All chunks processed so far are written. Unprocessed chunks are saved with `hitl_status: "pending"`. A summary is printed: `Session interrupted. N accepted, M edited, P flagged, Q pending.`

## Feature 5 — Export & Versioning

**Modules:** `ingestor/exporter.py`, `ingestor/corrections.py`

### What it does
Writes each reviewed chunk as an atomic `.md` file with YAML frontmatter. Generates a `manifest.json` mapping all chunks to source PDF page ranges. Maintains `corrections.json` as an append-only record of HITL actions.

### Acceptance Criteria

**AC-5.1 — Atomic file output**
Each chunk is written as a separate `.md` file at:
```
{processed_root}/{project_name}/{chunk_id}.md
```
`project_name` is derived from the source PDF filename (stem, lowercased, spaces replaced with underscores). File writes are atomic: write to a `.tmp` file, then rename. A partial write must never produce a corrupt `.md` file.

**AC-5.2 — YAML + content structure**
Each `.md` file begins with the YAML frontmatter block (between `---` delimiters), followed by a blank line, followed by the Markdown content. No other structure.

**AC-5.3 — Manifest generation**
After all chunks are exported, write `manifest.json` to the project output directory with this structure:
```json
{
  "source_file": "original.pdf",
  "project_name": "original",
  "generated_at": "ISO-8601 timestamp",
  "total_chunks": 42,
  "chunks": [
    {
      "chunk_id": "chunk_001",
      "file": "processed/original/chunk_001.md",
      "page_range": [1, 3],
      "hitl_status": "accepted",
      "confidence_score": 0.91
    }
  ]
}
```

**AC-5.4 — Corrections log (append-only)**
Every HITL action that is not a plain Accept is appended to `corrections.json` as a record:
```json
{
  "record_id": "uuid-v4",
  "chunk_id": "chunk_007",
  "timestamp": "ISO-8601",
  "action": "edited",
  "original_yaml": { ... },
  "corrected_yaml": { ... },
  "reason": "User-supplied string or null"
}
```
The file is a JSON array. On first write, initialize the array. On subsequent writes, append. Never overwrite existing records unless `--overwrite-corrections` CLI flag is present.

**AC-5.5 — Output directory safety**
If the output directory for a project already exists, do not overwrite existing `.md` files unless `--force` CLI flag is passed. Print a `WARNING` listing the files that would be overwritten and halt, waiting for `--force`.

## CLI Entry Point

**File:** `main.py`

```
Usage: python main.py <pdf_path> [--project <name>] [--force] [--overwrite-corrections]

Arguments:
  pdf_path                  Path to the source PDF (required)
  --project <name>          Override the derived project name
  --force                   Overwrite existing output files
  --overwrite-corrections   Allow overwriting corrections.json records
```

The pipeline runs in this order:
1. Load and validate `config.yaml`
2. Extract PDF → `DocumentContent`
3. Chunk → list of `Chunk` objects
4. Generate metadata for each chunk (Ollama)
5. HITL review loop
6. Export all chunks + manifest

Any unhandled exception at any stage must print a clean error message (not a traceback, unless `--debug` flag is present) and exit with a non-zero status code.
